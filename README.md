# Zara Challenge — Smartphones

A small catalogue application built against the _Tienda de móviles_ API. Users
can browse the first 20 smartphones, search by brand or name, open a product
detail page to pick a colour and storage variant, and manage a persistent
cart.

## Stack

| Area | Choice | Why |
| --- | --- | --- |
| Framework | React 19 + TypeScript (strict) | Current stable; brief requires ≥17 |
| Build tool | Vite 6 | Native ESM dev, concatenated + minified production build out of the box — no custom bundler config needed to satisfy the dev/prod modes in the brief |
| Routing | React Router v7 | `createBrowserRouter`, lazy-loaded routes |
| State | Context API + `useReducer` | Per the brief. Two concerns are split: cart state lives in a dedicated context; product data is colocated in the pages that use it (no single mega-context) |
| Styling | SCSS Modules + CSS custom properties | CSS variables are the brief's optional bullet; modules scope styles per component |
| Fonts | `Helvetica, Arial, sans-serif` | Per the brief |
| Tests | Vitest + React Testing Library + jsdom | Shares Vite's transform pipeline |
| Quality | ESLint 9 flat config, `jsx-a11y`, Prettier, Husky + lint-staged | Accessibility is a hard requirement, so the a11y plugin is non-negotiable |
| Icons | `lucide-react` | Tree-shakeable, zero runtime config |

## Getting started

```bash
nvm use              # Node 18
npm install
npm run dev          # dev mode (assets unminified, HMR)
```

The app boots on `http://localhost:5173` and works out of the box — the API
URL and key from the brief are baked in as defaults.

If you want to point at a different deployment of the API, copy the example
env file and edit it:

```bash
# macOS / Linux
cp .env.example .env

# Windows (cmd)
copy .env.example .env

# Windows (PowerShell)
Copy-Item .env.example .env
```

Vite picks `.env` up automatically on the next `npm run dev`.

### Key design decisions

**Split state, not one mega-context.** A single context holding `products`,
`cart`, and `searchTerm` together would re-render the entire tree on every
keystroke. Cart state is in its own context; product data is fetched by the
pages that render it via custom hooks.

**Pure cart reducer.** `cartReducer` is a plain function with no side effects,
which makes it trivial to unit-test without mounting React. The `lineId` is
generated by the provider rather than inside the reducer so the reducer stays
deterministic.

**Each "Add to cart" creates a new line.** The brief says "remove individual
products" (not "decrement quantity"), so the model doesn't carry a quantity
field — adding the same variant twice creates two lines, each with its own
`lineId`.

**localStorage hydration without races.** The cart state is lazily initialised
from `localStorage` via `useReducer`'s init argument (synchronous read on first
render), and a post-hydrate effect persists changes. A `hasHydratedRef` guards
against the initial-mount write that would otherwise clobber storage on boot.
Storage reads validate the shape of the blob — malformed data (hand-edited
localStorage, schema drift) is treated as "no prior state" rather than crashing.

**Abort-safe fetching.** Both `useProducts` and `useProduct` own an
`AbortController` that's torn down on unmount or when their inputs change. If
a user types quickly, the in-flight request for the previous query is aborted,
preventing out-of-order responses from overwriting the UI.

**Cold-start-tolerant timeout.** The API is hosted on Render's free tier,
which spins down after ~15 minutes of inactivity and takes 30–60s to wake up
on the first request. The HTTP client defaults to a 60s timeout and composes
the internal timeout signal with any caller-provided `AbortSignal` via
`AbortSignal.any`.

**Debounced search.** The list page holds the raw input value in local state
and derives a debounced value (300ms) that's passed to `useProducts`. This
keeps the input responsive while firing the API at a reasonable rate.

**Server-side filtering.** The brief explicitly requires filtering via the
API, so search is sent as a query parameter; nothing is filtered client-side.

## Accessibility

- All interactive elements are keyboard-reachable with visible focus states.
- Form controls have associated labels (`useId` for the search input,
  `fieldset`/`legend` for variant selectors).
- Variant selectors use `aria-pressed` to communicate toggle state to
  assistive tech.
- Results counter is an `aria-live="polite"` region so screen readers
  announce changes as the user types.
- Cart count in the navbar has an accessible name that includes the count.
- `prefers-reduced-motion` is honoured in the reset stylesheet.
- ESLint is configured with `jsx-a11y` so regressions surface at lint time.

## Testing

Two layers of tests are included as examples of the pattern:

- `cartReducer.test.ts` — pure-function coverage of the reducer's transitions
  and selectors.
- `useDebouncedValue.test.ts` — hook behaviour with fake timers.

With more time, I'd add:

- An integration test per page using MSW to stub the API surface.
- A cart persistence test mounting `CartProvider` twice to verify round-trip
  through `localStorage`.

## Known limitations

- **API key exposure.** The brief instructs the client to send the
  `x-api-key` header directly. The key is therefore visible in every
  browser's network inspector. In production this should sit behind a
  lightweight backend-for-frontend that injects the header server-side; the
  client would call the BFF instead of the public API.
- **No total-count in the list response.** The `/products` endpoint returns
  only the array, so the results counter reflects items returned, not a
  total available count. The brief only asks for 20 items, so no real
  pagination is implemented.
- **No offline fallback.** If the API is unreachable on first load, the list
  view shows an error state. A service-worker cache of recently-viewed
  products would be a sensible next step.

## What I'd do next

- **MSW-backed integration tests** for each view.
- **Error boundary** at the route level rather than per-page status handling.
- **Image preloading** on `mouseenter` of `ProductCard` to make detail
  navigation feel instant.
- **Skeleton loaders** tuned to the card/detail dimensions rather than a
  generic "Loading…" message.
- **BFF** to remove the client-side API key (see Known limitations).
